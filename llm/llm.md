某大厂 AI 应用
开发线上面试
1．你现在工作几年了？
> 根据个人实际情况回答。一般按实际工作年限说，比如"5年"。如果是转行或跨方向，可以说"做后端开发3年，其中AI方向1年"。
2．之前去过哪几家公司？
> 按实际经历说，突出与岗位相关的经验。例如："之前在XX公司做后端开发，在YY公司开始接触AI应用开发。"
3．做过哪些项目？
> 挑选2-3个最具代表性的项目，按 STAR 原则简要描述：项目背景、你的角色、技术方案、成果。优先说跟 AI 相关的。
4．你做过哪些 AI 相关项目？
> 例如智能客服、知识库问答、文档理解、AI Agent 等。说清楚项目规模和你的职责。

5．你是偏后端开发，还是偏 AI 应用开发？
> "偏 AI 应用开发"更匹配这个岗位。可以补充说"有扎实的后端基础，现在主要做 AI 应用层的开发"。

6．你对 AI 应用开发的理解是什么？
> AI 应用开发不只是调 API，而是把大模型能力落地到具体业务场景中。核心包括：需求拆解（判断什么场景适合用 AI）、RAG 流程设计（知识库检索 + 模型生成）、Prompt Engineering、Agent/Tool Calling、效果评估与持续优化。本质上是在模型能力之上构建可靠的业务系统。

7.AI应用开发和传统后端开发有什么区别？
> 传统后端：确定性的 CRUD、事务、状态机，输入输出可预期，逻辑可穷举。
> AI 应用：非确定性输出，依赖模型能力，需要处理幻觉、格式不稳定、推理成本高。多了 Prompt、RAG、Agent、向量检索等新组件。核心挑战从"逻辑正确"变成"输出可靠"。

8．你平时主要用哪些大模型？
> GPT-4o（通用能力强）、DeepSeek（代码和推理强）、Qwen（中文好、性价比高）、Claude（长上下文和代码能力好）。根据场景选模型。

9. OpenAI 、通义千问、文心一言、智谱DeepSeek 接触过哪些？
> OpenAI（GPT 系列）、通义千问（Qwen 系列）、DeepSeek 都用过。文心一言和智谱主要了解 API 能力，实际项目中用得少。

10．你是直接调用模型 API ，还是做过私有化部署？
> 两者都做过。原型验证和低延迟场景直接调 API；对数据安全要求高的政企项目，做过 DeepSeek 和 Qwen 的私有化部署（vLLM + 4090/A100）。

11．智能客服系统是你最近做的项目 ？
> 假设是。可回答："对，最近一个项目是某业务的智能客服系统，覆盖售前咨询和售后问题处理。"

12．智能客服系统整体架构是什么？
> 接入层（用户渠道：Web/App/企微）→ 网关层（鉴权、限流、路由）→ 业务层（意图识别、RAG检索、Agent调度）→ 模型层（大模型 API / 私有化部署）→ 数据层（向量库、对话记录、知识库文档）。

13．用户提问后，完整处理链路是什么？
> 用户提问 → 意图分类/路由 → 检索知识库（向量+关键词混合召回）→ Rerank 排序 → 拼接 Prompt → 调用大模型生成回答 → 流式返回给用户 → 记录日志。如果 Agent 场景则多一步工具调用。

14．智能客服是直接调用大模型，还是结合知识库？
> 必须结合知识库。直接调用大模型无法保证回答的准确性和时效性，企业场景下未经知识库约束的"自由发挥"是不可接受的。

15．为什么需要知识库，不直接问大模型？
> ① 大模型知识有截止日期，无法覆盖最新业务信息；② 存在幻觉，企业场景需要可追溯的答案来源；③ 无法覆盖私有业务数据（内部文档、用户手册等）；④ 知识库可以精确控制回答边界。

16.RAG的核心流程是什么？
> Indexing：文档解析 → 切片 → Embedding → 存入向量库。
> Retrieval：用户问题 → Embedding → 向量检索 + 关键词检索 → 混合召回 → Rerank。
> Generation：召回结果 + 用户问题 → 拼入 Prompt → 调用 LLM 生成回答。

17．你们的 RAG 流程怎么设计？
> 多路召回（向量检索 + BM25 关键词检索）→ Rerank 精排 → 动态 Prompt 组装（根据意图选择不同模板）→ LLM 生成 → 后处理（格式校验、敏感词过滤）。

18．文档是怎么接入知识库的？
> 用户上传/系统导入 → 文件格式识别 → 文档解析（提取文本）→ 切片（按策略分块）→ Embedding 向量化 → 存入向量库 + 元数据存入关系数据库。

19.PDF、Word、Excel、网页数据怎么处理？
> PDF：PyMuPDF/pdfplumber 提取文本，表格用 camelot/tabula。
> Word：python-docx 提取。
> Excel：openpyxl/pandas 逐行读取。
> 网页：Jina Reader / FireCrawl / BeautifulSoup 爬取清洗。

20．文档解析失败怎么处理？
> ① 分类记录失败原因（格式不支持、文件损坏、加密）；② 通过消息队列重试（最多3次）；③ 重试仍失败则标记为"解析失败"，通知管理员手动处理；④ 对图片型PDF接入 OCR 降级方案。

21．文档切片策略怎么设计？
> 按层级切：先按章节/标题分割，再对长段落按 chunk size 切。保留标题层级作为 metadata，便于检索时提供上下文。

22. chunk size 和 overlap 怎么设置？
> 经验值：chunk size 300-800 tokens，overlap 50-150 tokens。
> 具体取决于文档类型：代码/表格类用小 chunk（300-500），叙事类用大 chunk（500-800）。overlap 一般 10%-20%。

23．不同类型文档会用不同切分策略吗？
> 会。代码文档按函数/类分割，表格按行+表头描述切，QA 文档按一问一对切，长文按 Markdown 标题层级递归切。

24．文本是怎么转成向量的？
> 调用 Embedding 模型（如 text-embedding-3-small、bge-large-zh、m3e）将文本转为浮点数向量，维度通常 768/1024/1536。

25. embedding 模型用的哪个？
> 通常用 bge-large-zh-v1.5（中文效果好）或 text-embedding-3-small（OpenAI，通用能力强）。国产可选 m3e-base。

26.向量数据库用过哪些？
> Milvus（生产级，适合大规模）、FAISS（原型/小规模）、Elasticsearch（已有 ES 可复用）、Pinecone（云原生）。

27．项目里用的是什么向量数据库？
> Milvus 或 ES。Milvus 专门做向量检索性能好，ES 适合已有 ELK 体系且需要文本+向量混合搜索的场景。

28.Milvus、Faiss、Pinecone、Elasticsearch 向量检索了解吗？
> Milvus：分布式向量库，支持 GPU 加速和多种索引（IVF_FLAT、HNSW）。Faiss：Meta 开源的向量检索库，单机高效，适合小规模。Pinecone：全托管云服务，免运维但成本高。ES：通过 dense_vector 字段支持向量检索，优势是能与文本检索统一。

29．向量检索 topK 一般设置多少？
> 10-30。先取 topK=20-50 做粗召回，再通过 Rerank 取 top 3-5 作为最终上下文。

30. topK 设置太大或太小有什么问题？
> 太小：召回不全，可能漏掉相关文档，模型无法获取足够信息。
> 太大：引入大量噪声，超出上下文窗口，降低回答质量，增加 Rerank 和 LLM 推理成本。

31．相似度阈值怎么设置？
> 先收集一批测试数据，画出相似度分布曲线。一般阈值设为 0.6-0.8（余弦相似度）。低于阈值的检索结果丢弃，避免基于不相关内容的错误回答。

32．向量检索不准怎么办？
> ① 换更好的 Embedding 模型；② 混合检索（向量+关键词）；③ 调整 chunk 策略；④ 加 Rerank；⑤ 检查文档质量本身是否足够好；⑥ 使用多向量（每个段落生成多个向量，覆盖不同角度）。

33．只做向量召回够吗？
> 不够。向量检索对语义相似好，但精确关键词匹配（如产品型号、编号）弱。需要结合 BM25 关键词检索做互补。

34．有没有做关键词检索和向量检索混合召回？
> 有。常见方案：向量检索和 BM25 各自召回 topK，然后合并去重，用 Rerank 统一排序。权重可配，一般语义查询偏向量，精确查询偏 BM25。

35.BM25和向量检索有什么区别？
> BM25：基于词频和逆文档频率的精确关键词匹配，适合短文本、术语查询，速度快，不依赖模型。
> 向量检索：语义相似度匹配，能理解同义词和上下文，但需要 Embedding 模型，计算成本高。
> 两者互补，配合使用效果最好。

36. rerank 模型用过吗？
> 用过。例如 bge-reranker-v2、Cohere Rerank。Rerank 对粗召回结果逐对计算相关性分数，排序精度比向量检索高很多。

37．为什么需要 rerank ?
> 向量检索的相似度不一定是"问答相关性"，Rerank 模型专门计算 query 和 passage 的相关性，能显著提升 top 结果的准确率，减少噪声。

38．召回结果怎么拼进 prompt ?
> 按照"参考文档"格式拼接：注明文档标题、来源，用明确的分隔符区分不同片段。例如：
> ```
> 请根据以下参考资料回答用户问题：
> [1] 标题：xxx  内容：...
> [2] 标题：xxx  内容：...
> ```
> 并在 prompt 末尾强调"如果参考资料中没有相关信息，请如实告知"。

39．多个文档片段内容冲突怎么办？
> ① 在 prompt 中要求模型指出冲突并综合判断；② 引入来源优先级（如"高优先级文档" vs "低优先级文档"的字段）；③ 按文档更新时间排序，取最新的；④ 对于关键冲突，降级到人工处理。

40．知识库更新后，向量数据怎么同步？
> 增量更新：只有新增/修改的文档才重新生成向量。全量更新：定期（如每晚）做一次全量同步。实践中增量为主，全量兜底。

41．删除文档后，向量库怎么处理？
> 删除文档时同步删除对应向量（通过文档 ID 关联）。向量库不支持按 ID 删时，维护"已删除文档"表，检索时过滤。

42．用户问题知识库里没有，怎么回答？
> Prompt 明确要求：知识库找不到时必须说"没有找到相关答案"，不能编造。同时兜底："已记录问题，稍后人工跟进"。

43.AI回答错误或者幻觉怎么兜底？
> ① Prompt 约束（只基于知识库）；② 相似度阈值过滤；③ 敏感词/规则拦截；④ 人工审核开关；⑤ 用户反馈机制。

44．有没有做人工审核或人工转接？
> 有。置信度低、检测到敏感词、或用户不满意时自动转人工。完整对话上下文随转接传递。

45. Prompt 是你自己写的吗？
> 是的，我负责整个 Prompt 设计和迭代。设计时结合业务场景和模型特性反复测试优化。

46.怎么设计 system prompt ?
> ① 角色定位（你是谁）；② 行为边界（能/不能做什么）；③ 回答规范（格式、语气）；④ 知识库使用方式；⑤ 安全约束。持续根据 badcase 迭代。

47. system prompt 、 user prompt 、 assistant prompt 有什么区别？
> System Prompt：系统级行为规则，对话中通常固定。User Prompt：用户输入。Assistant Prompt：模型回复，可作为 few-shot 示例传到后续轮次。

48．如何让模型只基于知识库回答？
> System prompt 强调"只能根据提供的参考资料回答"。User prompt 只放检索内容，不加模型知识。温度调低（0-0.3）。

49．如何减少模型胡说八道？
> ① 温度 0-0.3；② Prompt 强调"不知道就说不知道"；③ 要求引用来源编号；④ 知识库阈值过滤；⑤ 输出校验；⑥ RAG 优先。

50．如何让模型按固定 JSON 格式输出？
> System prompt 给出 JSON Schema 和示例。要求只输出 JSON。设置 response_format={type:'json_object'}。温度设为 0。

51．模型输出 JSON 不合法怎么办？
> ① 用正则/JSON parser 提取 JSON 部分；② 失败则重试（加更严约束）；③ 准备默认兜底 JSON；④ 用 json-repair 库修复。

52. prompt太长怎么优化？
> ① 裁剪无关历史；② 用摘要替代完整历史；③ 精简 system prompt；④ 减少知识库片段（降低 topK）；⑤ 缩小 chunk。

53. 上下文窗口不够怎么处理？
> ① 摘要压缩历史；② 只保留最近 N 轮；③ 降低 topK 和 chunk size；④ 滑动窗口丢弃最早消息；⑤ 换长上下文模型（Claude 200K、GLM-4-128K）。

54. 多轮对话上下文怎么管理？
> 维护消息列表：system prompt + 历史摘要 + 最近 N 轮完整对话 + 本轮问题 + 检索结果。组装后传给模型。

55. 历史消息全部传给模型吗？
> 不。全部传会很快填满上下文。通常保留最近 5-10 轮，更早的用摘要压缩后传入。

56. 怎么做对话摘要？
> 每隔 N 轮（如 5 轮），让模型对前面对话生成摘要。将摘要作为 system prompt 的一部分，替代原始完整历史。增量更新。

57. prompt注入攻击了解吗？
> 了解。用户通过输入嵌入恶意指令，试图让模型忽略规则或泄露敏感信息。例如"忽略之前的指令"。

58. 用户故意让模型忽略规则怎么办？
> ① System prompt 尾部加强约束；② 输入过滤拦截注入关键词；③ 用户输入与指令明确隔离；④ 风险场景做权限分级。

59. 怎么防止敏感信息泄露？
> ① 输出过滤（手机号、身份证等正则）；② 知识库权限隔离；③ System prompt 禁止输出特定信息；④ 输出脱敏后处理；⑤ 日志脱敏。

60. 你了解 Agent 吗？
> 了解。Agent 是能自主理解任务、调用工具、执行多步推理的 AI 系统。核心组件：LLM + 工具集 + 记忆 + 规划能力。

61. Agent 和普通 ChatBot 有什么区别？
> ChatBot：单轮问答，无工具调用，被动回复。Agent：多步推理，主动规划，可调用外部工具（查数据库、调接口、联网搜索），有记忆和状态管理。
62. 做过工具调用吗？
> 做过。在 Agent 场景中，模型通过 function calling 调用外部 API，比如查订单、查天气、调用业务系统接口。

63. function calling 用过吗？
> 用过。OpenAI 的 function calling、DeepSeek 的 tool use 都对接过。需要在 API 调用时传入 tools 定义（name、description、parameters）。

64. 模型怎么决定调用哪个工具？
> 模型根据用户问题语义和工具描述（name + description）自动匹配。描述写得越清晰，模型选得越准。也可以通过 system prompt 限定场景。

65. 工具调用失败怎么处理？
> ① 捕获异常后让模型重新调用（最多重试 N 次）；② 返回错误信息给模型，让其给出兜底文案；③ 超时则降级（走无工具模式回答）；④ 记录失败日志用于监控。

66. 多工具调用怎么编排？
> 串行：第一步的输出作为第二步的输入，适合依赖链。并行：多个独立工具同时调用，适合查多个数据源。动态编排：模型根据中间结果决定下一步调用哪个工具（ReAct 模式）。

67. Agent出现死循环怎么办？
> ① 设置最大迭代次数（如 10 轮），超限强制退出；② 检测重复调用（工具+参数相同）则打断；③ 给模型提供"停止推理"的终止工具；④ 超时熔断。

68. 工具返回结果太长怎么办？
> ① 截断（保留前 N 字符）；② 摘要（让模型对工具返回做压缩）；③ 只提取关键字段传给模型；④ 分页返回。

69. 做过联网搜索吗？
> 做过。搜索作为 Agent 的一个 tool，通过搜索 API 获取实时信息，结果摘要后传入模型。

70 接过数据库查询工具吗？
> 接过。模型根据用户问题生成 SQL/查询条件，执行后返回结果给模型，由模型转换成自然语言回答。

71 接过业务系统接口吗？
> 接过。常见的有查订单、查物流、查库存。需要先定义接口的 OpenAPI Schema，注册为 tool。

72 用户一句话让AI查订单、改状态，怎么保证安全？
> ① 查询类操作可以放行，修改类操作必须用户二次确认；② 权限校验（用户只能查自己的订单）；③ 操作审计日志；④ 敏感操作走人工审批。

73 AI能不能直接操作生产数据？
> 绝对不能。所有写操作必须走业务接口（API），不能直连数据库。接口层面做权限校验和操作审批。

74 工具调用前要不要做权限校验？
> 必须做。每个工具调用前校验用户身份和权限，确保用户有权执行该操作和访问对应数据。

75 Agent执行结果怎么审计？
> 记录每条 Agent 执行轨迹：用户输入、模型推理过程、工具调用（参数+返回）、最终输出。存入日志系统用于回溯和分析。

76 模型API调用流程是怎样的？
> 应用层组装请求（messages + tools）→ 调用模型 API → 解析响应 → 如果是 function_call 则执行工具 → 将工具结果返回给模型继续生成 → 直到 final answer 返回给用户。

77 流式输出怎么实现？
> 设置 API 参数 stream=True，模型通过 SSE 逐 token 推送。后端将 chunk 逐片转发给前端，前端逐字渲染。

78 SSE和WebSocket用过吗？
> 都用过。SSE 适合服务端到客户端的单向流式推送（AI 对话场景），WebSocket 适合双向实时通信。

79 为什么AI对话一般用流式返回？
> 用户体验好——逐字展示降低用户等待感知。首 token 延迟毫秒级即可开始展示，避免白屏等待。

80 模型响应很慢怎么优化？

81. 首 token 延迟怎么优化？
> ① 精简 prompt（减少 system prompt 和历史）；② 知识库内容只传最相关的 topK；③ 用推理更快的模型；④ vLLM 部署时开启 prefill 优化；⑤ 保持服务端长连接减少握手开销。

82. 请求超时怎么处理？
> 设置合理超时时间（如 30s）。超时后：① 返回兜底文案给用户；② 异步重试（最多 1-2 次）；③ 记录超时日志；④ 熔断（连续超时 N 次后降级到备用模型）。

83. 模型接口失败怎么降级？
> ① 重试（指数退避，最多 3 次）；② 切备用模型（如 GPT 失败切 DeepSeek）；③ 降级到规则引擎或静态 FAQ；④ 返回友好提示。

84. 多模型怎么切换？
> 通过配置中心（配置模型名、API Key、endpoint）。运行时根据策略切换：主模型失败/超时/限流时自动切备用模型。

85. 做过模型路由吗？
> 做过。根据场景路由：简单问答走轻量模型（Qwen-Turbo），复杂推理走强模型（GPT-4o/DeepSeek），代码场景走代码模型。

86. 不同场景怎么选择不同模型？
> 简单问答（FAQs）：Qwen-Turbo/DeepSeek-Lite（快、便宜）。复杂推理：GPT-4o/DeepSeek-R1。代码生成：Claude/GPT-4o。长文档：Claude 200K/GLM-4-128K。中文：Qwen/DeepSeek。

87. GPT、DeepSeek、Qwen 这类模型怎么取舍？
> GPT-4o：通用能力最强但贵，适合核心场景。DeepSeek：代码和推理强，性价比高。Qwen：中文好，便宜，适合国内场景。按延迟/成本/质量 三者权衡。

88. 怎么控制模型调用成本？
> ① 轻量模型处理简单问题（分流）；② 缓存命中率高的场景；③ 设置 token 上限；④ 控制历史轮数；⑤ 批量调用有折扣。

89. token 消耗怎么统计？
> 在 API 调用日志中记录每次的 prompt_tokens、completion_tokens、total_tokens。按用户/场景/模型汇总，监控趋势。

90. 用户频繁调用怎么限流？
> 用户级别限流（如 10 次/分钟），接口级别限流（如 100 QPS）。超出返回 429，前端提示"请求太频繁"。

91. 同一个问题会不会做缓存？
> 会。对于高频问题（如"如何退货"），缓存完整回答。对相似问题做语义缓存（向量检索命中缓存的回答）。

92. AI 回答结果适合缓存吗？
> 适合。对于确定性知识（政策、流程、产品参数），缓存是安全的。对于时效性强的信息（价格、库存），缓存时间要短或不缓存。

93. 哪些场景适合缓存？
> ① 静态知识问答（政策、FAQ、帮助文档）；② 高频重复问题；③ 模板化回复；④ 不需要实时数据的工作流。不适合：个性化问题、实时数据查询、需要推理的复杂问题。

94. 模型返回内容怎么做日志记录？
> 记录请求 ID、用户 ID、问题、模型响应、token 消耗、延迟、模型名。日志不可修改，用于分析、审计和效果评估。敏感数据脱敏后存储。

95. 企业知识库怎么做权限隔离？
> 文档级别打标签（部门/角色/安全等级），向量检索时通过 metadata 过滤。检索结果再经过业务层做二次权限校验。

96. 不同部门的数据怎么隔离？
> 文档入库时标注部门 ID，向量检索时传入用户所属部门 ID 过滤。部门间的数据互相不可见。

97. 用户只能查自己有权限的文档，怎么实现？
> 文档元数据中保存权限列表（如用户/角色/部门）。检索时拼接权限过滤条件，只返回用户有权限访问的文档。

98. 向量检索时怎么带权限过滤？
> 向量库支持 metadata filter（Milvus 的 expr、ES 的 post_filter、Pinecone 的 filter），在检索时传入权限条件，只搜索用户可见的文档。

99. 如果向量库不支持复杂权限怎么办？
> ① 先按权限缩小搜索范围（从业务层获取用户可见文档 ID 列表，传给向量库做 ID 范围过滤）；② 检索后做二次过滤（在业务层剔除无权限结果）；③ 按权限分区存储（不同部门用不同 collection/namespace）。

100. 敏感信息怎么脱敏？
> 文档入库时自动检测并脱敏（正则匹配手机号、身份证号、银行卡号等，用 *** 替代）。AI 回答侧再做一次输出过滤，确保无明文敏感信息。

101. 用户上传文件怎么做安全校验？
> ① 文件类型校验（白名单：pdf/docx/txt）；② 文件大小限制（如 < 10MB）；③ 病毒扫描；④ 内容安全检测（涉政/涉黄/涉恐）；⑤ 解析后检查是否包含恶意脚本。

102. AI 回答中出现敏感数据怎么办？
> ① 输出过滤器（正则匹配 + 模型判断）；② 敏感词命中时拦截回答或打码处理；③ 记录告警，人工审核；④ 定期评估模型输出质量。

103. 模型服务调用外部接口，有没有数据泄露风险？
> 有。用户问题可能包含敏感信息，传给外部模型 API（如 OpenAI）有数据泄露风险。解决方案：① 使用私有化部署的模型；② 调用外部 API 时做脱敏预处理；③ 签署数据保护协议。

104. 私有化部署大模型的意义是什么？
> ① 数据不出域，满足合规要求；② 低延迟（无需公网传输）；③ 可定制（微调、量化）；④ 长期成本可控（高频场景比 API 调用便宜）。

105. 金融、保险、政企场景用 AI 最大的风险是什么？
> 合规风险（错误回答导致法律后果）、数据安全（敏感信息泄露）、不可控性（模型幻觉/偏见）、监管要求（可解释性、审计要求）。

124. 接口幂等怎么保证？
> 请求方传唯一请求 ID（幂等键），服务端根据幂等键去重。相同请求只执行一次，返回相同结果。用 Redis 记录已处理的幂等键。

125. 分布式锁用过吗？
> 用过。基于 Redis（Redisson/SetNX）或 ZooKeeper 实现。用于防止重复执行任务，如文档向量化任务。

126. Redis 在 AI 项目里一般用来做什么？
> ① 缓存（热门回答、文档内容、Session）；② 限流计数器；③ 分布式锁；④ 对话上下文临时存储；⑤ 幂等键去重。

127. Redis 缓存哪些数据？
> ① 高频问题的 AI 回答；② 用户 Session 和对话状态；③ 知识库文档元数据；④ Embedding 向量（小规模场景）；⑤ 限流计数器。

128. Redis 缓存和数据库怎么保证一致性？
> 先更新数据库，再删除/更新缓存（Cache Aside 模式）。允许短时间不一致的用 TTL 自动过期。一致性要求高的场景不做缓存。

129. 消息队列在 AI 项目里怎么用？
> ① 文档解析/向量化任务异步处理；② 削峰填谷（突发请求排队）；③ 日志采集和上报；④ 多服务解耦（知识库同步、索引更新）。

130. RocketMQ / Kafka 用过吗？
> 用过。Kafka 适合高吞吐日志和大数据场景，RocketMQ 适合业务消息（事务消息、延时消息），各有优劣。

131. 文档解析、向量化任务为什么适合异步？
> 耗时长（一个文档可能需要几秒到几十秒）、资源密集（CPU/内存/API）、不需要用户实时等待。异步处理可以批量操作、并发执行、失败重试。

132. 数据库主要用 MySQL 还是 PostgreSQL？
> 两者都可。PG 支持 JSONB、数组、全文检索等，更适合知识库场景。MySQL 生态成熟，运维成本低。按团队熟悉度选。

133. AI 对话记录怎么设计表结构？
> 主表 conversations：id, user_id, title, created_at, updated_at。
> 明细表 messages：id, conversation_id, role, content, tokens, created_at。
> 需对 conversation_id 和 user_id 建索引。

134. 知识库文档表怎么设计？
> documents：id, title, file_type, file_url, status, creator_id, department_id, created_at, updated_at。
> chunks：id, document_id, content, chunk_index, metadata（JSON）。
> 外键关联 documents.id → chunks.document_id。

135. 向量 ID 和业务文档 ID 怎么关联？
> 向量库中每条向量记录包含字段：vector_id、doc_id、chunk_id、content、metadata。通过 doc_id 关联业务表。向量删除/更新时通过 doc_id 筛选。

136. 数据库慢 SQL 怎么优化？
> ① EXPLAIN 分析执行计划；② 加合适索引；③ 优化查询条件（避免 % 前缀模糊、避免 OR）；④ 分页优化（游标分页替代 OFFSET）；⑤ 读写分离。

137. 索引加多了有什么影响？
> ① 写入变慢（每次写要更新索引）；② 占用更多磁盘空间；③ 查询优化器可能选错索引。需要在查询性能和写入性能之间平衡。

138. 查询日志和调用日志怎么存？
> 查询日志（用户提问、检索过程、命中结果）：存 ES（便于全文检索和分析）。调用日志（API 调用、token 消耗、延迟）：存 ClickHouse 或时序数据库（便于聚合统计）。

139. 怎么判断 AI 客服是否真的提升效率？
> 对比接入 AI 前后的指标：人工客服接听量变化、平均响应时间、用户满意度、问题解决率。看是否减少了人工处理量。

140. AI 客服准确率怎么评估？
> 用测试集（人工标注的 Q&A pair）评测模型回答的正确率。结合自动评估（BLEU/Rouge/LLM-as-Judge）和人工抽样评估。

141. 命中率、召回率、满意度这些指标怎么看？
> 命中率 = 知识库能回答的问题占比（越高越好）。召回率 = 正确文档被检索出来的比例（RAG 的核心指标）。满意度 = 用户点赞/差评比例。

142. 业务方说 AI 回答不准，你怎么排查？
> ① 复现问题；② 查看实际检索到的文档片段是否正确（召回问题？）；③ 看 prompt 是否限制了正确回答（指令问题？）；④ 看模型输出是否符合预期（模型问题？）。记录 badcase 逐一分析。

143. 是模型问题、知识库问题，还是 prompt 问题，怎么判断？
> 先看知识库：检索到的文档内容是否正确？如果不正确 → 知识库/检索问题。如果正确但模型回答不对 → 模型/prompt 问题。再用相同输入换更强模型测试 → 如果强模型回答正确，说明模型能力问题；如果都不对，说明 prompt 设计问题。

144. AI 应用上线前怎么测试？
> ① 单元测试（工具调用、接口逻辑）；② 集成测试（完整 RAG 链路）；③ 标注测试集评测（准确率/召回率）；④ 红队测试（安全/Prompt 注入）；⑤ 灰度上线（小流量验证）。

145. 做过问题集评测吗？
> 做过。收集一批标准 QA 对，用自动化脚本批量测试模型回答，自动对比预期结果和实际结果，生成评测报告。

146. 人工标注数据怎么用？
> ① 作为测试集评测模型效果；② 标注"好回答"作为 few-shot 示例；③ 标注 badcase 用于 prompt 优化；④ 部分标注数据用于模型微调。

147. 怎么持续优化知识库问答效果？
> ① 收集 badcase 建立反馈闭环；② 定期更新知识库（新增/过期文档）；③ 优化切片和检索策略；④ 迭代 Prompt；⑤ A/B 测试验证效果。

148. 你觉得 AI 应用项目最难的地方是什么？
> 不是技术本身，而是效果评估和持续优化。AI 输出不确定，很难像传统软件一样"测试通过就上线"。如何建立有效的评测体系、收集 badcase、针对性优化，是最大的挑战。

149. 你怎么看 AI 应用开发未来的发展？
> ① 从 ChatBot 到 Agent 化（工具调用、自主规划）；② 多模态（文本+图片+语音）；③ 模型能力更强，应用层会更薄，但系统工程（可靠性、可观测性、安全性）会更重要；④ 垂直领域深度定制会是竞争壁垒。
